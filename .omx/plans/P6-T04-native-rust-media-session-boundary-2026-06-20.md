# P6-T04 - Native Rust Media Session Boundary

## Requirements Summary

Source: Multica PER-55 / Phase 6 two-person voice chat production path.

Acceptance criterion: `start_native_voice_media_session` requires a valid backend voice join/session and returns native media proof without fake state. A caller-supplied session id or peer id is not enough; the command must derive authority from backend voice state, OpenMLS media exporter state, and signed runtime peer metadata.

Relevant paths:
- `apps/desktop/src-tauri/src/lib.rs`: Tauri voice commands, native Rust media proof builder, signed runtime peer derivation, and backend tests.
- `crates/media/src/*`: Rust capture, Opus, SFrame, and transform bridge primitives used by the Tauri proof path.
- `.omc/plans/discrypt-plan.md`: original voice/media invariants requiring OpenMLS/SFrame-backed protected media.
- `docs/release/handoff-2026-06-10-current-state.md`: current release reset and backend-truth claim boundary.

## Implementation Steps

1. Derive the native voice media local/remote peer boundary from the joined group session's signed runtime peers and current backend-governed local role.
2. Reject missing sessions, unjoined/stopped sessions, and caller-provided peer ids that do not match backend-derived runtime peers.
3. Keep the native media proof generated by the existing Rust Opus/SFrame pipeline and OpenMLS media exporter secret.
4. Add focused backend tests for missing session, unjoined session, stale/non-backend peer ids, and the valid success path.
5. Run targeted Tauri backend tests and formatting; classify evidence as local backend/Tauri harness evidence, not production two-machine voice proof.

## Risks And Mitigations

- Risk: UI or tests could pass arbitrary peer ids and create optimistic voice media proof.
  Mitigation: compare request peer ids against backend-derived signed runtime peers before generating SFrame output.
- Risk: stale voice session state could survive after leave/restart.
  Mitigation: keep `session.joined` and session id checks before peer derivation and test unjoined rejection.
- Risk: provider signaling could be mistaken for media relay.
  Mitigation: proof remains Opus/SFrame media carried through backend text/control signaling; providers remain signaling-only.
- Risk: this task could expand into audio device selection or two-machine media signaling.
  Mitigation: leave those out of scope; only strengthen the command boundary and backend tests.
