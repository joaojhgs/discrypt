# P12-T01 Playwright UX Matrix Plan

## Requirements Summary
- Expand `apps/ui/tests/e2e/*` with explicit Playwright coverage for first-run/profile setup, group creation, invites, manual and automatic admission, text, voice UI, config, and member panel behavior.
- Keep assertions backend-evidence driven: local-dev fallback command state is acceptable harness evidence, but the test must not claim production remote delivery, admitted membership, online presence, or active media beyond command/browser evidence.
- Preserve the existing Discord-like shell and avoid runtime UI changes unless a missing selector blocks coverage.

## Acceptance Criteria
- A single PER-96 matrix spec exercises setup, config, automatic-admission group creation, invite metadata, local text send status, manual admission approval/refusal, member-panel route/presence state, voice join/leave, denied microphone behavior, and hidden diagnostics/manual pairing controls.
- Matrix assertions prove success from command-backed state surfaces such as admission policy, admission requests, message `data-message-state`, member panel route copy, and voice test IDs.
- Release evidence records commands, artifacts, and the evidence level as frontend/local harness evidence, not production split-machine/Tauri WebDriver proof.

## Implementation Steps
1. Add `apps/ui/tests/e2e/ux-matrix.spec.ts` using existing Playwright patterns and `support/app-shell.ts`.
2. Seed only local-dev fallback command state inside the test when a backend-governance state variant is required for review/presence/route assertions.
3. Add `docs/release/per96-playwright-ux-matrix-2026-06-26.md` with changed files, verification commands, artifact locations, and known evidence limits.
4. Run `npm --prefix apps/ui run test:e2e`; if the full suite exposes a narrow issue, run and report the targeted spec as additional evidence.

## Risks and Mitigations
- Risk: fixture-only success could hide missing backend truth. Mitigation: all stateful assertions must read from UI surfaces generated by command state, and docs must call the result frontend/local harness evidence only.
- Risk: full e2e suite duration or existing unrelated flakes. Mitigation: preserve exact command output in the issue comment and add targeted spec evidence if needed.
- Risk: seeded localStorage shape can drift. Mitigation: keep the seed narrow and validate visible UI outcomes rather than internal fields alone.

## Verification
- Required: `npm --prefix apps/ui run test:e2e`.
- Supplemental if needed: `npm --prefix apps/ui run typecheck`, `npm --prefix apps/ui run build`, or targeted `npx playwright test tests/e2e/ux-matrix.spec.ts --workers=1`.
