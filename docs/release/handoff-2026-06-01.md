## What Is Still Missing

### P1c — Real-time Message Delivery (effort: ~1 day)

**Problem:** The frontend polls `poll_app_events` every 5 seconds (`main.tsx:384–397`). This means:
- Messages from a remote peer arrive up to 5 s late
- The transport pump pushes frames through the DataChannel, but the receiving side only reflects them on the next poll tick

**Fix:**
1. **Backend** (`lib.rs`): After `send_message` or `receive_text_delivery_envelope` appends to `state.events`, call `app_handle.emit("app_event", payload)` (requires passing `AppHandle` into the command via Tauri v2 state injection)
2. **Frontend** (`main.tsx`): Replace / augment the `setInterval` with `await listen("app_event", handler)` (Tauri `@tauri-apps/api/event`)
3. Remove or extend the polling interval to 30 s as a fallback keep-alive

**Files to touch:** `lib.rs` (emit after mutations), `main.tsx` (listen subscription), `commands.ts` (no change needed)

---

### P2 — Voice Audio Pipeline (effort: ~3–5 days)

**Problem:** `join_voice` records device permissions and updates UI state, but zero audio flows between users. The mic is requested (`getUserMedia`), but the stream is only used to compute RMS activity for the speaking indicator — it is never added to any `RTCPeerConnection`.

**What exists:**
- `requestVoiceDeviceAccess()` in `main.tsx:229` — acquires mic stream, computes RMS
- `voiceAccess.activity_rms_i16` piped to backend — speaking indicator works
- `VoicePanel` renders participant list, mute toggle, volume slider, join/leave buttons
- Mute: sets `self_muted` state in backend — never calls `audioTrack.enabled = false`

**What must be built:**

1. **WebRTC audio negotiation**
   - Create `RTCPeerConnection` per voice session on join
   - `stream.getTracks()` → `peerConnection.addTrack(track, stream)`
   - Re-run offer/answer exchange (via existing signaling transport) with audio codec included
   - Mute: `audioTrack.enabled = !muted` on the local track

2. **Remote audio output**
   - `peerConnection.ontrack` → create `<audio autoPlay>` element per remote participant
   - One `<audio>` per `RTCRtpReceiver` / participant

3. **Speaking detection**
   - `RTCRtpReceiver.getSynchronizationSources()` → audio level → `speaking` boolean per participant (poll at ~100 ms)

4. **Multi-participant mesh**
   - Current model is 1:1 DataChannel. Voice with >2 participants needs either:
     - SFU (Selective Forwarding Unit) — recommended for scale
     - Full mesh (N*(N-1)/2 connections) — simpler for small groups (≤4)
   - For MVP: full mesh is fine; SFU is a future gate

**Files to touch:** `main.tsx` (RTCPeerConnection lifecycle, `<audio>` elements, speaking poll), `lib.rs` (optional: emit voice-session events), `commands.ts` (optional: `VoiceAudioState` type)

**Problem:** Group creation has no adapter picker or ICE server fields. All groups use hardcoded Google STUN (`stun:stun.l.google.com:19302`) with no way to configure TURN or choose MQTT vs Nostr vs IPFS as the signaling adapter.

**Fix:**
1. Add step 2 to `CreateGroupPanel` — adapter radio (MQTT / Nostr / IPFS) + optional custom endpoint field
2. STUN/TURN text inputs (default pre-filled with Google STUN)
3. Pass these to `create_group` request; backend already has the adapter selection logic
4. Group settings modal (gear icon) with same fields for existing groups

**Files to touch:** `main.tsx` (`CreateGroupPanel` component), `commands.ts` (`CreateGroupRequest` type, add `adapter_kind` + `ice_servers` fields), `lib.rs` (`create_group` handler — wire the new fields)

### P4 — Credentialed TURN / Public Relay (effort: variable, infra work)

Not a code gap — requires provisioning:
- A TURN server (coturn or equivalent) with credentials
- Or a TURN SaaS credential (Twilio, Metered, etc.)
- Wire credentials into the ICE server config from P3

Without TURN, peer connections only work when both peers are on the same LAN or have public IPs. NAT traversal fails for the majority of real-world users.
